## 0. 重新认识图引擎：什么是“确定性工作流”？

在 JoySafeter 的底层架构中，当你画出的画布上**没有任何一个节点开启了 `useDeepAgents`** 时，后端的 `GraphBuilderFactory` 就会将你的图交由 `LanggraphModelBuilder` 引擎进行编译。

**这一模式的本质是“控制流优先”**：
- **拓扑是死的**：你拉的每一根线就是代码里的一个绝对分支（Edge）。执行流只可能按照你画死的路线运行。
- **状态是共享的**：所有节点共享一个全局的状态字典 `state.context`，上游节点往里写数据，下游节点从中读数据。
- **适用场景**：要求**绝对可控、步骤明确、容错率低**的工业级流水线（例如必须按 “信息收集 -> 漏洞扫描 -> 报告生成” 严格执行的合规审计）。

## 1. 核心路由与边（Edge）的控制魔法

在标准的 LangGraph 构建器中，不仅有用来干活的 Agent，更重要的是**控制走向的特殊连线和逻辑节点**。理解这三种边，就能掌握工业级图构建的精髓：

### 1.1 顺序执行（`normal` 边）
- **触发**：从一个 Agent 的右侧直接连线到另一个的左侧。
- **意义**：强同步。前置节点的工作必须彻底完成（状态数据写入 `state.context`），后置节点才会被唤醒。

### 1.2 条件分流（Condition / Router 节点 + `conditional` 边）

Agent 执行完了，面临岔路口怎么办？这就需要 **Python 条件表达式** 来介入。

**实战案例：风险风控告警分流**
假设上游的 [安全分析 Agent] 识别完漏洞后，将危险等级写进了状态中。
1. 拖入一个 **Condition 节点**，连接在分析 Agent 之后。
2. 在该节点的 **条件表达式 (Condition Expression)** 框内编写符合 Python 语法的验证逻辑：
   ```python
   # 拦截高危漏洞走向专属告警通道
   state.get('context', {}).get('severity') in ['Critical', 'High']
   ```
3. 这个节点会散发出两根线（True 分支和 False 分支）。在连出的时候，系统会弹出侧边栏让你设定 `route_key`（路由键）分别为 `"true"` 和 `"false"`，这根线会被染成蓝色，这就是带逻辑的边。

> 💡 **进阶：多路 Router**
> 如果有超过两条岔路（比如“Web漏洞”、“物理漏洞”、“社交漏洞”三种工单派发），请使用 **Router 节点**。它可以按从上到下的顺序匹配多条 Python 表达式（正则表达式、字典取值等皆可），匹配到了，流就会顺着对应的标签边滑向对应的处理 Agent。

### 1.3 重试与迭代（Loop Condition + `loop_back` 边）

大模型总是无法一次生成对格式，或者你需要持续探测直到达成条件，该怎么做死循环内的受控迭代？

1. 拖入 **Loop Condition 节点**，它可以配置 **最大迭代次数（Max Iterations）**。
2. 配置终止条件，例如：`state.get('context', {}).get('is_code_clean') == True`
3. 关键动作是**连回边（`loop_back`）**：从这个条件节点拉一根线，**逆向连接回**它前面的那个执行 Agent。
   - 此时，这根线会变成一根**醒目的紫色虚线**，标志着这是一个合法的环（Cycle）。
   - 如果达到最大迭代次数依然条件不通过，它会自动抛出超时错误，防止流水线彻底死锁。

## 2. 把事情做快：并行执行与聚合结果 (Fan-Out & Fan-In)

在真实业务场景中，“资产测绘”阶段通常需要同时进行子目录爆破、端口扫描、WHOIS 查询，如果串行执行将极其浪费时间。

### 2.1 扇出（Fan-Out 向下并行）
- 你可以从同一个 `START` 节点或前置节点，**同时拉出三根线**分别指向 `[端口扫描 Agent]`、`[指纹识别 Agent]` 和 `[子域名搜集 Agent]`。
- 图引擎 `LanggraphModelBuilder` 在编译时一旦发现同源的多条边，会自动启动 `asyncio` 异步并发执行这三个节点，极致压榨并发性能。

### 2.2 扇入与聚合（Fan-In / Aggregator）
- 多线程跑完了数据，怎么汇总写出一份总报告？
- 你必须在所有并发节点的末尾，**统一汇聚连接到一个 [Aggregator（聚合器）] 节点**。
- **Aggregator 的核心参数 `error_strategy`**：
  - 设置为 `best_effort`（尽力而为）：即使某个扫描挂了（比如超时），Aggregator 也会把剩下两个成功的结果拼在一起放进全局状态往下传，不会阻断主流程。
  - 随后，这股汇流的线再连给一个 `[汇总报告专家 Agent]`。它就能在 `state.context` 中一次性获得三份并发扫描的结果了。

## 3. 标准图模式的排错与调试（The Runner）

画完图后怎么保证它能稳定运转不崩溃？请善用画布右侧边界拉出的 **Runner（调试器）** 控制台。

1. **Schema Check (校验 JSON)**：不要等运行才发现你忘记配 `systemPrompt` 或者画出了一条死胡同无头线。点击工具栏中央的 ✅ 按钮，校验器会立刻提示图中的致命错误。
2. **State Variable Panel (全局状态内窥镜)**：在 Runner 中投递第一句话触发执行后，点击右上角的调试面板，你可以**即时、肉眼看清**当前 `state.context` 这个全局大字典里流淌的每一个 JSON 字段和它们的值。（这是定位为什么 Condition 节点分支走错了的最快手段：看一看上游到底把参数存成了什么拼写）。
