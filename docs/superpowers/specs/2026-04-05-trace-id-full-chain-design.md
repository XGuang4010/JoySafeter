# Trace ID Full-Chain Propagation Design

## Problem

The backend has three independent trace ID systems that never connect:

1. **HTTP `trace_id`** — generated in `LoggingMiddleware`, stored on `request.state`, echoed as `X-Trace-Id` header. Only exists for HTTP requests, not WebSocket.
2. **StreamState `trace_id`** — generated per streaming turn in `stream_event_handler.py`. Flows into SSE events and `ExecutionTrace` persistence. Never linked to the client's `request_id` or the HTTP trace.
3. **Langfuse trace** — auto-generated by `LangfuseCallbackHandler`. Cannot be correlated with internal traces.

Additionally, `AgentRun.trace_id` is declared but never written, and resume turns don't mirror events to `AgentRunEvent`.

This creates 7 identified gaps where trace context is lost:

1. HTTP `trace_id` never reaches WebSocket path
2. Client `request_id` never linked to `StreamState.trace_id`
3. `StreamState.trace_id` absent from LangGraph `RunnableConfig`
4. `AgentRun.trace_id` column always NULL
5. Langfuse traces not correlated with internal traces
6. A2A calls generate independent `request_id`
7. Resume turns don't mirror events to `AgentRunEvent`

## Solution: contextvars-based Propagation

A single `ContextVar[str]` carries the trace_id through the entire async call chain. Set once at the request entry point, readable everywhere via `get_trace_id()`.

```
Client request_id ─────────────────────────────────────┐
    │                                                    │
    ▼                                                    │
set_trace_id(request_id)  [WS handler / HTTP middleware] │
    │                                                    │
    ├── loguru auto-injects trace_id in every log line   │
    ├── StreamState.trace_id = get_trace_id()            │
    ├── LangGraph config["configurable"]["trace_id"]     │
    ├── Langfuse CallbackHandler(trace_id=...)           │
    ├── A2A client sends X-Request-ID header             │
    │                                                    │
    ▼                                                    │
SSE events carry trace_id ──────────────────────────────┤
AgentRun.trace_id = get_trace_id()                      │
ExecutionTrace.id = trace_id                            │
AgentRunEvent.trace_id = trace_id (from SSE mirror)     │
                                                         │
Client correlates: request_id == trace_id in all outputs ┘
```

## Component Design

### 1. Trace Context Module

**New file: `backend/app/core/trace_context.py`**

```python
"""Trace context propagation via contextvars."""

import uuid
from contextvars import ContextVar

_trace_id_var: ContextVar[str] = ContextVar("trace_id", default="")


def set_trace_id(trace_id: str | None = None) -> str:
    """Set trace_id for the current async context.

    Always normalizes to a valid UUID string. If the input is not a
    valid UUID (e.g., a free-form client request_id), a new UUID is
    generated instead. Returns the trace_id.
    """
    tid = _normalize_to_uuid(trace_id) if trace_id else str(uuid.uuid4())
    _trace_id_var.set(tid)
    return tid


def get_trace_id() -> str:
    """Get the current trace_id (empty string if not set)."""
    return _trace_id_var.get()


def _normalize_to_uuid(value: str) -> str:
    """Return value if it's a valid UUID string, otherwise generate a new UUID."""
    try:
        return str(uuid.UUID(value))
    except (ValueError, AttributeError):
        return str(uuid.uuid4())
```

Design choices:
- Default is empty string, not None — avoids null checks downstream.
- `set_trace_id()` always normalizes to a valid UUID string. Client `request_id` may be free-form, but `AgentRun.trace_id` and `ExecutionTrace.id` are UUID columns. If the client sends a non-UUID, a new UUID is generated.
- `set_trace_id()` returns the ID for convenience at call sites.
- asyncio `create_task()` automatically copies contextvars to child tasks.

### 2. Entry Points

#### 2a. HTTP Middleware — `app/common/logging.py`

```python
from app.core.trace_context import set_trace_id

async def dispatch(self, request, call_next):
    trace_id = request.headers.get("X-Request-ID") or str(uuid.uuid4())
    set_trace_id(trace_id)                    # ← inject into contextvars
    request.state.trace_id = trace_id         # keep for backward compat
    log = logger.bind(trace_id=trace_id, ...)
    ...
```

#### 2b. WebSocket — `websocket/chat_ws_handler.py`

When a WS frame is received and parsed into a `ChatStartFrame`:

```python
from app.core.trace_context import set_trace_id

# In the frame dispatch method, BEFORE create_task() — context is captured at task creation
set_trace_id(frame.request_id or None)
```

**Important:** `set_trace_id()` must be called in the WS handler's receive loop, **before** `ChatTaskSupervisor.create_task()` is called. `asyncio.create_task()` copies the current context at creation time, so the trace_id must be set before the task is spawned.

#### 2c. WebSocket — `websocket/chat_turn_executor.py`

`execute_standard_turn()` receives `request_id` as a parameter. Add at the top as a safety net (for any code path that calls the executor directly):

```python
from app.core.trace_context import set_trace_id

async def execute_standard_turn(self, request_id, ...):
    set_trace_id(request_id)  # ← safety net; primary injection is in 2b
    ...
```

**Why both 2b and 2c?** Section 2b is the primary injection point (before `create_task()`). Section 2c is a defensive safety net — if `execute_standard_turn()` is ever called from a different code path without going through the WS handler, the trace_id still gets set.

### 3. loguru Auto-Injection

In `setup_logging()`, replace the existing `logger.configure(extra=...)` call (line 89 of `logging.py`) with a version that includes a patcher. This ensures every log line carries the current trace_id without explicit `logger.bind()`:

```python
from app.core.trace_context import get_trace_id

def setup_logging():
    logger.configure(
        patcher=lambda record: record["extra"].update(
            trace_id=get_trace_id() or record["extra"].get("trace_id", "-")
        ),
        extra={"trace_id": "-", "method": "-", "path": "-", "client": "-"},
    )
    ...
```

The patcher runs per-log-call and reads from contextvars. If no context is set (e.g., during startup before any request), falls back to the existing extra value or "-".

### 4. StreamState

**File: `app/utils/stream_event_handler.py`**

```python
from app.core.trace_context import get_trace_id

class StreamState:
    def __init__(self, thread_id: str):
        self.trace_id = get_trace_id()  # ← read from context, don't generate
        ...
```

No changes to `_build_sse_envelope()` — it already reads `state.trace_id`.

### 5. LangGraph Config

**File: `app/api/v1/chat.py` — `get_user_config()`**

```python
from app.core.trace_context import get_trace_id

config: RunnableConfig = {
    "configurable": {
        "thread_id": thread_id,
        "user_id": str(user_id),
        "trace_id": get_trace_id(),   # ← new
    },
    "recursion_limit": 300,
    "callbacks": get_langfuse_callbacks(enabled=settings.langfuse_enabled),
}
```

Graph nodes and tools can read `config["configurable"]["trace_id"]` if needed (e.g., for A2A calls), but most code should use `get_trace_id()` directly.

### 6. Langfuse Integration

**File: `app/core/agent/langfuse_callback.py`**

```python
from app.core.trace_context import get_trace_id

def get_langfuse_callbacks(enabled: bool):
    if not enabled:
        return []
    trace_id = get_trace_id()
    handler = CallbackHandler(
        trace_id=trace_id if trace_id else None,
        trace_name="graph_execution",
    )
    return [handler]
```

`CallbackHandler` supports `trace_id` parameter. When set, the Langfuse trace uses this ID, making internal `ExecutionTrace.id` and Langfuse trace directly joinable.

### 7. A2A Client

**File: `app/core/a2a/client.py`**

Add trace context to outbound HTTP headers for cross-service correlation. The JSON-RPC `id` field remains a per-request UUID (it serves request/response correlation, not distributed tracing):

```python
from app.core.trace_context import get_trace_id

# Keep existing: request_id = str(uuid.uuid4()) for JSON-RPC id
# Add to outbound HTTP headers:
headers["X-Request-ID"] = get_trace_id() or request_id
```

### 8. Persistence

#### AgentRun.trace_id

**File: `app/services/run_service.py` — `create_run()`**

```python
from app.core.trace_context import get_trace_id

tid = get_trace_id()
run = AgentRun(
    ...
    trace_id=uuid.UUID(tid) if tid else None,
)
```

Since `set_trace_id()` always normalizes to a valid UUID string, `uuid.UUID(tid)` will never raise ValueError.

#### ExecutionTrace

No change needed — `_persist_trace_data()` already reads `StreamState.trace_id`, which now comes from contextvars.

#### AgentRunEvent

No change needed — `_mirror_run_stream_event()` reads from the SSE payload, which already carries the correct trace_id.

#### Resume Turn Fix

**File: `app/websocket/chat_turn_executor.py`**

The resume turn path must:
1. Call `set_trace_id()` at the start (same as standard turn)
2. Pass `agent_run_id` and `assistant_message_id` to `_send_event_from_sse()` so events are mirrored to `AgentRunEvent`

For `agent_run_id` sourcing: the resume turn should look up the existing `AgentRun` for the conversation's current thread_id (the run that was interrupted), or create a new child run linked to the original. Implementation will determine which approach based on the `ChatTaskEntry` structure at that point.

## Assumptions and Constraints

- **Starlette version >= 0.36.2** required for `BaseHTTPMiddleware` to correctly propagate `ContextVar` through `call_next()`. Earlier versions had a known bug.
- **No `run_in_executor()` or `ProcessPoolExecutor`** in the codebase (verified). If added in the future, trace context must be manually copied.
- **`asyncio.create_task()` copies context automatically** (Python 3.7+). LangGraph's async execution uses this mechanism, so trace_id propagates through graph nodes and tool calls.
- **`langfuse >= 2.0`** required for `CallbackHandler(trace_id=...)` parameter support.
- **Trace ID format is UUID v4.** Not compatible with OpenTelemetry's 32-hex-char trace ID format. Future OTel integration would need a mapping layer.

## Verification

1. **Unit test**: Set `set_trace_id("test-123")`, verify `get_trace_id()` returns it, verify it propagates through `asyncio.create_task()`. Also test that non-UUID input gets normalized to a valid UUID.
2. **Patcher test**: Call `set_trace_id("...")`, then `logger.info("hello")` — verify the log output contains `trace_id=...` even without explicit `logger.bind()`. This is the core loguru integration.
3. **Integration test**: Send a WS message with `request_id: "abc..."` (valid UUID), verify:
   - All SSE events have `trace_id` matching the normalized UUID
   - `AgentRun.trace_id` in DB matches
   - `ExecutionTrace.id` in DB matches
   - Langfuse trace (if enabled) uses the same ID
4. **Log grep**: `grep trace_id=<uuid> logs/app.log` should show all log lines for that request.
5. **Resume turn**: Verify resume turns also produce events with correct trace_id and mirror to `AgentRunEvent`.
6. **BaseHTTPMiddleware**: Verify `set_trace_id()` in middleware dispatch is readable inside a downstream route handler via `get_trace_id()`.

## Non-Goals

- OpenTelemetry integration (future iteration)
- Frontend trace visualization UI
- Trace-based sampling or rate limiting
- Merging AgentRun and ExecutionTrace tables (separate concern)
